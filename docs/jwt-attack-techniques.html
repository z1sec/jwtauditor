JWT攻击技术

安全指南 Reading time: 12 min

JSON Web Token（JWT）广泛用于身份验证和授权，但如果实现不当，它们可能会受到各种攻击。本指南探讨了常见的JWT攻击技术、其工作原理以及如何防范。

重要提示： 本指南仅供教育目的。请使用此知识保护您的应用程序安全，而不是攻击他人。

常见JWT攻击技术

以下是针对JWT实现最常见的攻击技术：

算法混淆

利用未验证头部中指定算法的实现。

None算法

将算法设置为"none"以绕过签名验证。

弱密钥

暴力破解用于签名令牌的弱HMAC密钥。

令牌篡改

修改有效载荷以更改用户角色或权限等声明。

令牌重放

在令牌应过期后重复使用捕获的令牌。

密钥ID注入

操纵"kid"参数以强制使用特定密钥。

JWK头部注入

在"jwk"头部参数中嵌入恶意公钥。

JWK集URL注入

操纵"jku"参数以从攻击者控制的URL获取密钥。

签名绕过

利用未正确验证签名的实现。

X.509证书注入

通过"x5c"头部参数注入自签名证书。

内容类型操纵

更改"cty"参数以启用XXE或反序列化攻击。

目录遍历

在"kid"参数中使用路径遍历来访问敏感文件。

算法混淆攻击

How It Works

算法混淆（也称为算法替换）攻击利用了未正确验证JWT头部中指定算法的实现。攻击涉及将算法从非对称算法（如RS256）更改为对称算法（如HS256）。

1

服务器使用RS256（RSA）算法，用私钥签署令牌并用公钥验证它们。

2

攻击者获取公钥（通常公开可用）。

3

攻击者将头部中的算法从RS256更改为HS256。

4

攻击者使用HS256算法和公钥作为密钥签署令牌。

5

如果服务器不验证算法，它将使用公钥作为HMAC密钥来验证令牌。

Example Attack

// 原始JWT头部 (RS256)
{
  "alg": "RS256",
  "typ": "JWT"
}

// 修改后的JWT头部 (HS256)
{
  "alg": "HS256",
  "typ": "JWT"
}

// 然后攻击者使用HS256算法和公钥作为密钥签署令牌

预防措施

  • 始终验证算法并拒绝具有意外算法的令牌
  • 使用白名单方法仅接受特定算法
  • 根据算法实现严格的密钥类型检查
  • 使用设计上防止算法混淆的库

None算法攻击

工作原理

"none"算法攻击利用了接受带有"none"算法的令牌的实现，这表示令牌未签名。如果服务器接受此类令牌，攻击者就可以伪造令牌而无需任何密钥。

1

攻击者获取一个有效的JWT并对其进行解码。

2

攻击者修改头部，将"alg"设置为"none"。

3

攻击者按需修改有效载荷（例如，更改用户角色）。

4

攻击者创建一个带有空签名的新令牌或完全移除签名。

攻击示例

// 原始JWT头部
{
  "alg": "HS256",
  "typ": "JWT"
}

// 修改后的JWT头部
{
  "alg": "none",
  "typ": "JWT"
}

// 签名部分要么为空，要么完全移除
// 格式: header.payload.  （注意末尾的点后面没有内容）

预防措施

  • 明确拒绝使用"none"算法的令牌
  • 使用允许算法的白名单
  • 确保您的JWT库默认不接受"none"算法
  • 无论指定什么算法，都要始终验证签名

弱密钥攻击

工作原理

当JWT使用弱密钥通过HMAC算法（HS256、HS384、HS512）签名时，攻击者可以暴力破解密钥并伪造有效令牌。常见的弱密钥包括字典单词、短字符串或可预测的模式。

1

攻击者获取一个使用HMAC算法签名的有效JWT。

2

攻击者使用字典或暴力破解方法来猜测密钥。

3

对于每次猜测，攻击者尝试验证令牌的签名。

4

一旦找到正确的密钥，攻击者就可以使用任何有效载荷创建新令牌。

使用的工具

  • JWTAuditor密钥暴力破解器: 我们的工具，用于使用常见字典测试JWT密钥
  • Hashcat: 一款流行的密码破解工具，支持JWT破解
  • John the Ripper: 另一款支持JWT的密码破解工具

预防措施

  • 使用至少256位（32字节）的强随机生成密钥
  • 考虑使用非对称算法（RS256、ES256）而不是HMAC
  • 定期轮换密钥
  • 使用密钥管理器生成和存储强密钥

令牌篡改

工作原理

令牌篡改涉及修改JWT的有效载荷以更改用户角色、权限或身份等声明。只有当攻击者也能创建有效签名时，此攻击才可能实现，这可能通过算法混淆或弱密钥等其他攻击来实现。

攻击示例

// 原始JWT有效载荷
{
  "sub": "1234567890",
  "name": "John Doe",
  "role": "user",
  "iat": 1516239022,
  "exp": 1516242622
}

// 篡改后的JWT有效载荷
{
  "sub": "1234567890",
  "name": "John Doe",
  "role": "admin",  // 从"user"更改为"admin"
  "iat": 1516239022,
  "exp": 1516242622
}

预防措施

  • 使用强签名算法和密钥
  • 实现适当的服务器端声明验证
  • 不要仅依赖JWT声明进行关键授权决策
  • 考虑对敏感数据使用加密JWT（JWE）
  • 对敏感操作实现额外的服务器端检查

令牌重放攻击

工作原理

令牌重放攻击涉及捕获有效的JWT并在应该失效后重新使用它，例如在注销或过期后。这对于未适当跟踪令牌使用或未实现令牌撤销的系统来说尤其有问题。

1

The attacker intercepts a valid JWT (e.g., through network sniffing or XSS).

2

合法用户注销或其会话过期。

3

攻击者重复使用捕获的令牌访问受保护的资源。

预防措施

  • 为令牌使用较短的过期时间
  • 对已注销的令牌实施令牌黑名单
  • 使用JWT ID（jti）声明跟踪和撤销特定令牌
  • 实施刷新令牌轮换
  • 使用安全的HttpOnly cookie进行令牌存储以减轻XSS风险

密钥ID（kid）注入

工作原理

JWT头部中的"kid"（密钥ID）参数用于指示应使用哪个密钥来验证签名。如果此参数未得到适当验证，攻击者可以操纵它指向他们控制的密钥或利用路径遍历漏洞。

1

攻击者识别使用"kid"参数选择验证密钥的JWT实现。

2

攻击者修改"kid"参数以指向他们控制的密钥或利用路径遍历。

3

如果成功，服务器使用攻击者控制的密钥验证签名。

攻击示例

// 原始JWT头部
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "key1"
}

// 包含路径遍历的修改JWT头部
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "../../public/key"
}

// 指向已知弱密钥的修改JWT头部
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "weak_key"
}

预防措施

  • 验证和清理"kid"参数
  • 对允许的密钥ID使用白名单方法
  • 避免在"kid"参数中使用文件路径
  • 为密钥存储实施适当的访问控制
  • 使用具有安全轮换的密钥管理系统

JKU/X5U URL操纵

工作原理

JWT头部中的"jku"（JWK集URL）和"x5u"（X.509 URL）参数指向外部URL，服务器可以在那里获取用于签名验证的密钥。如果这些参数未得到适当验证，攻击者可以：

  1. 将URL指向托管恶意密钥的自己的服务器
  2. 使用URL进行服务器端请求伪造（SSRF）攻击
  3. 利用内部网络访问权限扫描或攻击内部系统

攻击示例

// 原始令牌头部
{
  "alg": "RS256",
  "typ": "JWT",
  "jku": "https://trusted-server.com/.well-known/jwks.json"
}

// 恶意令牌头部
{
  "alg": "RS256",
  "typ": "JWT",
  "jku": "https://attacker.com/malicious-jwks.json"
}

预防措施

  • 禁用通过jku/x5u参数进行动态密钥获取
  • 如果需要动态获取，则使用允许URL的白名单
  • 验证和清理所有外部URL
  • 为出站请求实施网络级限制
  • 使用缓存/固定密钥而非动态获取

权限提升

工作原理

JWT攻击中的权限提升涉及修改令牌声明以获得比原始授予更高的权限。这通常针对与授权相关的声明，如角色、权限和用户标识符。

  1. 识别JWT有效载荷中的授权声明（角色、管理员、权限等）
  2. 修改这些声明以授予更高权限
  3. 重新签名令牌（如有需要，使用其他攻击技术）
  4. 使用修改后的令牌访问受限资源

攻击示例

// 原始有效载荷
{
  "sub": "user123",
  "role": "user",
  "permissions": ["read"],
  "iat": 1516239022
}

// 提升后的有效载荷
{
  "sub": "user123",
  "role": "admin",
  "permissions": ["read", "write", "delete", "admin"],
  "iat": 1516239022
}

预防措施

  • 在信任声明之前始终验证令牌签名
  • 实现超越JWT声明的服务器端授权检查
  • 在令牌设计中使用最小权限原则
  • 定期审计和验证用户权限
  • 实施适当的会话管理和令牌验证
